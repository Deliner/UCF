# UCF capability and evidence matrix

This document is the canonical statement of what the current repository can
do. A source document being accepted by the parser is not evidence that UCF can
execute, verify, or formally prove the declared behavior. The current package
is a `0.1.x` production-preview Python control plane for CPython 3.12 on
Linux/x86_64 with no SLA; it is not a stable API. External adapters remain
experimental exact proofs. CAP-214 remains planned until the complete release
checklist is proved.

Status meanings:

- `implemented`: the exact scope in the row has an executable repository path
  and a reproducible test command. It makes no broader claim.
- `experimental`: some code or declaration support exists, but its contract,
  end-to-end evidence, or trust boundary is incomplete. It is not a stable
  support claim.
- `planned`: the capability is absent. The evidence cell names its
  dependency-ordered backlog owner.

UCF currently provides no formal verification. Generated verification methods
are obligations that user-owned code must implement; a passing generated suite
is evidence only for the assertions actually executed by that suite.

## Interpretation rules

Behavior-IR parser acceptance records a well-formed document only. It never
raises a claim from declared to observed, mapped, tested, or verified. Trust IR
evaluates independent claim predicates over exact immutable bases; it has no
numeric or mutable promotion chain. CAP-212 records and reassesses exact
`tested` evidence but never promotes it. Formal `verified` evidence is
unavailable in the current contract.

Historical design notes and proposals may describe a target larger than this
matrix. They are not current capability evidence. When a document conflicts
with this matrix, this matrix and executable commands above control the current
claim.
