# REL-001 end-to-end adoption benchmark

REL-001 is a bounded, reproducible release-evidence benchmark for UCF 0.1.0.
It is not a general ecosystem-support or stable-release claim. The canonical
machine-readable result is [`rel001-report.json`](rel001-report.json), whose
structural digest is
`4f0e77045ca5c0cf4994d3059585aefa549854eee2897c2b7968b35f1881854b`.

## Proven scope

Three ecosystem lanes run four frozen component roots from clean copies:

| Component | Transport | Source files / bytes | Inventory | Candidates | False candidates | Decisions A/E/R/U | Eligible / uncovered / debt | Tested / verified |
| --- | --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| Python legacy quote | none | 4 / 1,584 | 24 | 4 | 2 | 1/1/1/1 | 4/0/1 | 1/0 |
| TypeScript/Fastify quote | HTTP | 7 / 40,955 | 42 | 4 | 3 | 1/0/3/0 | 6/2/2 | 1/0 |
| Go stdlib quote | HTTP | 6 / 6,692 | 51 | 4 | 2 | 1/0/2/1 | 12/8/9 | 1/0 |
| Go stdlib platforms | CLI, event | 9 / 20,810 | 62 | 1 | 0 | 1/0/0/0 | 9/8/8 | 2/0 |

Totals are 26 files and 70,041 source bytes, 179 inventory records, 13
candidates, 7 false candidates, 13 explicit candidate decisions, one edited
ambiguity resolution, 5 materializations, 4 implementation mappings, and 5
fresh `tested` claims. `mapped` and `verified` remain zero because no separate
mapping-approval artifact or formal/exhaustive proof exists. Mapping approvals
and change approvals are both explicitly zero.

Here a false candidate is one rejected by the frozen review oracle or
materially edited before acceptance. An uncertain candidate is reported as
unresolved coverage debt, not silently counted as false or verified. The
decision sequence is scripted and reproducible; it measures review actions,
not human review duration or usability.

Every component establishes Ratchet 2.0 with separate behavior and coverage
ledgers. Its baseline and unchanged rerun outcome is
`pass_with_legacy_coverage_debt`. This means exact inherited uncertainty may
remain; it is not a project-wide clean pass. Added, changed, or reintroduced
debt blocks, while a comparable resolution becomes protected. Ratchet 1.0 is
unchanged.

## Execution evidence

The benchmark performs three repetitions. Each lane is copied, run through
its native check, the complete installed-wheel workflow, the native check
again, and an immutable-manifest recheck. All twelve workflow repetitions and
their native-before/native-after checks passed. Each selected adapter also
passes the exact 17-case conformance profile before measurement.

Median phase time in milliseconds:

| Lane | Copy | Native pre | UCF workflow | Native post | Manifest recheck |
| --- | ---: | ---: | ---: | ---: | ---: |
| Python | 0.295 | 15.902 | 765.324 | 16.421 | 0.002 |
| TypeScript/Fastify | 0.714 | 1,095.674 | 1,667.644 | 215.065 | 0.002 |
| Go HTTP | 0.676 | 214.309 | 915.493 | 214.744 | 0.002 |
| Go CLI/event | 0.878 | 214.315 | 1,416.918 | 214.794 | 0.003 |

These are three local samples, not latency guarantees. The exact samples,
minimums, and maximums are retained in the JSON report.

The run used CPython 3.12.13 on Linux/x86_64, Node 22.22.3, npm 10.9.8, and Go
1.26.5 linux/amd64. The report binds the wheel and dependency-lock digests,
exact installed distributions, adapter and conformance artifacts, four lane
drivers, compiler, scenario runner, build/source-contract helpers, and
toolchains. Only runtime samples are excluded from structural replay equality.

## Spec and evidence overhead

Accounting version 1.0.0 counts canonical top-level resources and separates
explicitly authored resources from derived UCF resources. Shared policy and
change-lifecycle resources are not duplicated across components.

| Component | Legacy bytes | Authored bytes / records | Derived bytes / records |
| --- | ---: | ---: | ---: |
| Python legacy quote | 1,584 | 4,976 / 1 | 178,502 / 13 |
| TypeScript/Fastify quote | 40,955 | 3,400 / 1 | 216,592 / 13 |
| Go stdlib quote | 6,692 | 3,455 / 1 | 287,725 / 13 |
| Go stdlib platforms | 20,810 | 1,741 / 1 | 316,621 / 19 |

Shared policy adds 550 authored bytes and one record. The archived Python
change lifecycle adds 1,763 authored bytes/one record and 17,330 derived
bytes/five records. Total overhead is 15,885 authored bytes in six records and
1,016,770 derived bytes in 63 records over 70,041 immutable legacy bytes.
Embedded resources make derived byte volume intentionally non-additive as a
human editing estimate; the per-component and shared buckets prevent double
allocation.

## Reproduction

Static validation checks the closed report codec and internal accounting:

```sh
uv run --locked --extra dev python tools/rel001_benchmark.py check \
  --report docs/benchmarks/rel001-report.json
```

Release replay rebuilds and installs the wheel, executes all four lanes three
times, compiles with that installed wheel, and compares every non-runtime
section to the checked report:

```sh
uv run --locked --extra dev python tools/rel001_benchmark.py verify-published \
  --report docs/benchmarks/rel001-report.json --repetitions 3
```

The replay is also the observable `rel001-benchmark` member of:

```sh
python3 tools/quality_gates.py --profile all
```

## Limitations

- The evidence covers these four frozen roots only and is not a population
  estimate for Python, TypeScript, Fastify, Go, HTTP, CLI, or event systems.
- Linux/x86_64 and the exact toolchain versions above are the verified
  platform boundary; Windows, macOS, other architectures, and other versions
  require requalification.
- Review decisions are scripted fixture oracles, not human review-time or
  usability evidence. Mapping and change approvals are absent and count zero.
- The event case is a local file-spool procedure, not a hosted broker,
  durability, ordering, delivery, or exactly-once guarantee.
- Adapters and checks run with the invoking user's authority and are not a
  sandbox. Digests provide traceability, not signatures or authenticity.
- The benchmark creates tested evidence only. `verified: 0`; no formal
  verification claim is made.
- REL-001 is experimental release evidence, not stable support. Compatibility,
  security, privacy, licensing, support, deprecation, and release policy remain
  the explicit REL-002 acceptance boundary.
